# Day Start — 量化研发方法论

> 本文档回答三个问题：
> 1. 量化交易的本质是什么？怎么思考量化？
> 2. 怎么从零研发一套量化软件？
> 3. 怎么根据你的想法和需求，生成一个可执行的量化路线计划？

---

## 一、量化交易的本质

### 1.1 一句话定义

**量化交易 = 用数学模型替代人的主观判断，做三件事：选什么、下多少、什么时候走。**

任何量化策略，不管多复杂，拆开就是这三个决策。你的 28 天路线图就是围绕这三件事搭建的。

### 1.2 赚钱逻辑先于代码

量化最容易犯的错误：上来就写代码，策略逻辑说不清楚。

**铁律**：写任何策略代码前，先填空——
> 「这个策略赚钱的原因是：_________________________________」

填不出来 → 不写代码。填得出来且逻辑自洽 → 继续。

你 28 天线路图里的 4 条策略，每条都有清晰的赚钱逻辑：

| 策略 | 赚钱逻辑 |
|------|---------|
| 趋势跟踪 | 市场有惯性——涨的会继续涨，直到趋势反转。用 EMA 交叉捕捉趋势起点，ATR 止损限制反转亏损 |
| 动量策略 | 双层时间框架确认——快动量 + 慢动量同向时，趋势延续概率更高 |
| 突破策略 | 价格突破关键价位后往往会加速——Donchian Channel 捕捉突破，成交量确认有效性 |
| 费率套利 | 现货 + 合约锁定价差，收取资金费率——不赌方向，赚的是市场结构钱 |

### 1.3 你赚的是什么钱？

每笔交易的利润可以拆成三层：

```
总利润 = Alpha（你的策略能力） + Beta（市场整体波动） + 噪声（运气）
```

- 趋势策略赚的是 **Beta + Alpha**（市场涨 + 你的选币和择时）
- 费率套利赚的是 **纯 Alpha**（锁定价差的套利空间，与市场方向无关）
- 均值回归赚的是 **噪声回归**（偏离均值的价格会回来——但不一定会回来）

**关键认知**：回测里 Beta 驱动的利润，实盘里可能恰好赶上熊市消失。要区分你赚的是什么钱。

## 二、量化的两层思维模型

### 2.1 策略层：选什么、下多少、什么时候走

这是每天交易时的三个决策。你的系统里对应：

| 决策 | 对应模块 | 开发日期 |
|------|---------|---------|
| 选什么（信号） | 策略引擎 (trend/momentum/breakout/funding_arb) | Day 3-4 |
| 下多少（仓位） | 仓位管理 (position_sizer.py) | Day 8 |
| 什么时候走（出场） | 分层止损 (stop_loss.py) | Day 9 |
| 什么情况下不交易 | 异常检测 + 熔断 (anomaly_detector.py) | Day 10 |

### 2.2 系统层：三圈防御

策略再好在失控的系统上跑也是送钱。系统层用三圈防御：

```
第一圈（日常，全自动）: 仓位管理 → 分层止损 → 异常检测 → 熔断
第二圈（周期，半自动）: 参数滚动优化 → 策略健康度评分 → 退役评估
第三圈（根基，人工）: 策略逻辑是否还成立？我能解释为什么赚钱吗？
```

你的 28 天路线里，Week 1-2 搭第一圈，Week 3 搭第二圈，Week 4 的 Dashboard 给第三圈提供可视化的依据。

### 2.3 数据层：四层金字塔

```
L1: 现货 K 线 + 深度（主信号）— Day 1 搞定
L2: 合约持仓量 OI + 资金费率（情绪）— Day 11 接口就绪
L3: BTC.D + 恐惧贪婪指数（宏观）— Day 11 接口就绪
L4: 链上 + 社交（辅助归因）— Day 19-21
```

越上层越精确但越窄，越下层越模糊但覆盖面越广。代码结构上每层独立采集，互不阻塞。

## 三、从想法到路线图：四步拆解法

这是生成 28 天冲刺路线的方法，你以后想开新项目时自己就能做。

### 3.1 第一步：画出终点

先定义"做完是什么样"，不要定义"怎么做"。

你的终点画像：
```
一个跑在东京服务器上的量化系统：
- 每天自动拉数据、跑策略、下订单
- 风控层层兜底，不会爆仓
- 赚钱/亏钱都能在 Dashboard 上解释清楚
- Telegram 推送到手机，不用天天看服务器
```

### 3.2 第二步：逆推必经阶段

从终点逆推，找出不可跳过的阶段。量化交易的阶段是固定的：

```
实盘赚钱
  ↑ 需要
部署 + 风控 (Week 2)
  ↑ 需要
策略回测验证可行 (Week 1 后半)
  ↑ 需要
数据管线 (Week 1 前半)
```

这四个阶段任何人做量化都要走一遍，无非是快慢问题。

### 3.3 第三步：每个阶段拆成天级任务

每天 「学 + 做 + 产出」三段式：

| 环节 | 说明 |
|------|------|
| 学 | 理解今天要解决的问题的本质（不是看教程，是理解为什么） |
| 做 | 写代码实现，必须有独立测试 |
| 产出 | 一个可运行的模块 + 一份学习笔记 |

比如 Day 8 仓位管理：
- 学：凯利公式的数学原理（为什么重仓必爆不是概率问题是数学问题）
- 做：position_sizer.py + 和 RiskManager 的集成
- 产出：仓位管理模块 + 学习笔记

### 3.4 第四步：标注依赖关系和风险

每项任务标注：
- 依赖什么（阻塞项）
- 有什么风险（如果 X 没到位怎么办）
- 降级方案（没有 API key 时用模拟数据）

这就是 28 天路线里每个 Day 末尾「基础设施依赖」表格的来源。

## 四、量化开发的工具箱设计原则

### 4.1 模块化：每块独立可测

```
不要写一个"大交易系统.py"
要写：position_sizer.py → 独立可测
      stop_loss.py       → 独立可测
      anomaly_detector.py → 独立可测
```

每个模块有自己的测试文件，改一个不会搞崩另一个。

### 4.2 版本快照：vX.Y.Z 目录

```
v0.1.0/  → MVP 雏形（回测 + 4 策略）
v0.2.0/  → 风控闭环（Day 8-11）
v0.3.0/  → 实盘就绪（Day 12-14）
```

每个版本是一个完整的可运行代码包，不依赖"等一下再补"的部分。

### 4.3 盲区检验：自动化安全网

你现在的 `_blind_spot_check.py` 检查 6 维 26 项。这套检查的框架可以复用到任何量化项目：
- 安全开关状态
- API 权限最小化
- 数据新鲜度
- 测试通过率
- 敏感文件排除
- 待办同步

## 五、遇到"不知道怎么做"时的决策框架

量化开发中反复出现的选择题，用这个框架决策：

### 5.1 先做多还是先做空？

**永远先做多。** 做多最大亏损 100%，做空理论亏损无上限。先验证多路径跑通（信号→风控→下单→日志），再做空。

### 5.2 用现成框架还是自己写？

| 场景 | 选择 |
|------|------|
| 回测引擎 | 现成（backtrader/vectorbt）→ 你自己选了自研，因为要控制事件循环细节 |
| 数据清洗 | 自己写（pandas 够用，不需要 ETL 框架） |
| Dashboard | 现成（Streamlit，不要自己写前端） |
| 策略基类 | 自己写（轻量，掌控事件循环和成本模型） |

原则：核心逻辑自己掌控，展示层用现成。

### 5.3 参数多少个合适？

**≤ 3 个。** 超过 3 个参数就是过拟合的开始。你 28 天路线里的趋势策略只有 fast_period、slow_period、atr_stop 三个参数，就是这个原因。

### 5.4 样本内/外怎么分？

**70/30 分割。** 样本外夏普 < 样本内夏普 × 0.7 → 过拟合嫌疑。你的 `parameter_scan.py` 已经内置了这个检查。

## 六、28 天冲刺路线的设计复盘

你的路线图之所以合理，因为遵循了这些原则：

1. **Week 1 (数据+策略)** → 先把轮子造出来
2. **Week 2 (风控+部署)** → 加刹车和保险带
3. **Week 3 (复盘+外部信号)** → 理解为什么赚/亏
4. **Week 4 (仪表盘+自动化)** → 把前三周的东西可视化、自动化

每周内部：
- 前 3 天：核心模块开发
- 第 4 天：外部数据/信号接入
- 第 5-6 天：验证/部署/加固
- 第 7 天：复盘

**为什么不是 14 天或 56 天？** 28 天 = 4 周，每周一个主题，每天 3-4 小时共 100-120 小时——刚好覆盖量化系统的最小可行版本。
