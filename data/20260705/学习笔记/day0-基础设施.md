# Day 0 学习笔记 — 基础设施与安全

## 一、为什么是东京云服务器，不是 VPN

### 1.1 延迟的链路分析

```
国内直连 → 币安 API：
  请求: 你的电脑 → 国内网络 → 国际出口 → 币安服务器
  延迟: 150-300ms
  问题: 国际出口拥堵时延迟波动大，高峰期可能 500ms+

VPN 挂东京 → 币安 API：
  请求: 你的电脑 → 国内网络 → VPN 服务器(东京) → 币安服务器
  延迟: 200-400ms
  问题: 多了一跳，而且 VPN 隧道本身有加密/解密开销

东京云服务器 → 币安 API：
  请求: 云服务器(东京) → 币安服务器(同在东京/新加坡)
  延迟: 2-10ms
  优势: 物理距离最短，没有中间商赚差价
```

### 1.2 「VPN 挂代理 ≠ 云服务器」

这是最容易被误解的点。VPN 的作用是**伪装你的位置**——你的代码还是在你自己的电脑上跑，网络路径反而更长。

云服务器的作用是**把你的代码搬到交易所隔壁**——代码和数据在同一个物理区域，延迟最小。

**量化交易拼的不是策略多聪明，是 200ms 延迟里你能不能比别人快 190ms。** 对于我们的低频日线策略来说延迟不重要，但这个基础设施认知是通用的——将来如果要做更实时的事情（订单簿分析、闪电贷套利），东京服务器是刚需。

## 二、交易所 API 安全

### 2.1 最小权限原则

这是安全领域最基础也最重要的原则：**只给你需要的权限，多的一个都不给。**

```
✅ 现货交易权限 — 策略需要下单
✅ 读取权限 — 策略需要查余额、查行情
❌ 提现权限 — 永远不开。开了 = 被黑就归零
❌ 转账权限 — 同上
❌ 合约交易权限（初期）— 先用现货跑稳，合约是下个月的课题
```

### 2.2 纵深防御

单靠 API 权限不够，需要多层防护：

| 层级 | 措施 | 防什么 |
|------|------|--------|
| 交易所侧 | IP 白名单绑定东京服务器 IP | 别人拿到你的 API key 也用不了 |
| 服务器侧 | API key 放在环境变量（.env），不入库 | 代码泄露 ≠ 密钥泄露 |
| 仓库侧 | .gitignore 排除 .env | 防止误提交到 GitHub |
| 资金侧 | 只放策略所需最低资金，盈利定期提现 | 极端情况损失可控 |

### 2.3 为什么盈利要定期提现

> 交易所账户里的币，不是你的币。只有提到你掌控私钥的钱包里，才是你的币。

FTX（2022）教会整个行业：交易所可以在一夜之间从「好好的」变成「提不了现」。你在交易所里的钱是 IOUs（我欠你），不是真正的资产。**盈利不提现 = 你在帮交易所保管你的利润。**

## 三、双交易所设置

### 3.1 不只是备份——是交叉验证

币安 + OKX 双账户的意义：

| 用途 | 说明 |
|------|------|
| 交叉验证 | 两个交易所的 K 线价差 > 0.5% → 数据异常，标记不交易 |
| 故障切换 | 币安宕机 → 策略自动切到 OKX 执行 |
| 套利侦察 | 价差持续扩大 → 可能有套利机会（跨交易所价差套利） |
| 风险分散 | 一个交易所出事，另一半资金还在 |

### 3.2 单一交易所的风险

- 交易所被黑（历史上发生过很多次）
- 交易所宕机维护（你策略正好要在这个时间点交易）
- 交易所冻结你账户（AML/KYC 问题）
- 交易所跑路（小交易所，但也要防）

**双交易所不是 over-engineering，是 basic survival。**

## 四、工具链选型

### 4.1 AI 模型的分工

| 模型 | 用途 | 月费 |
|------|------|------|
| Claude Sonnet（API） | 策略设计、报告生成、复杂分析 | ¥100-200 |
| DeepSeek（API） | KOL 情绪分析、新闻摘要（轻量任务） | ¥10-20 |

分工逻辑：
- Claude 做「需要深度思考」的事——策略逻辑审查、回测报告生成、异常诊断
- DeepSeek 做「量大但简单」的事——每天几十条推文的情绪分类、新闻标题摘要

**不要用一个模型做所有事。贵模型做重活，便宜模型做轻活。**

### 4.2 Telegram Bot 为什么是币圈标配

- 免费、API 简单、手机端开箱即用
- 支持交互指令（`/status` `/halt` 等）
- 币圈 KOL、交易所、项目方全在 Telegram 上——你不需要多装一个 App
- Bot 的推送触达率远高于邮件（谁还看邮件？）

Bark/PushDeer 作为备胎——Telegram 服务器被墙的时候，国内直连的推送不受影响。

### 4.3 回测框架的选择

backtrader vs vectorbt vs Freqtrade vs 自己写：

| 框架 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| backtrader | 事件驱动回测，策略逻辑复杂 | 灵活，社区大，文档全 | 速度中等 |
| vectorbt | 参数扫描，向量化回测 | 极快（几万组合秒出） | 只适合简单策略 |
| Freqtrade | 开箱即用的完整交易机器人 | 内置交易所适配、Telegram | 定制化受限 |
| 自己写 | **永远不要** | — | Bug 比你想象的多的多 |

我们选 backtrader 的原因：策略有入场/出场/止损/止盈的分支逻辑，需要事件驱动而不是向量化。

## 五、Git 仓库与配置分离

### 5.1 代码和配置为什么要分开

```
仓库里有的（可以公开）:
├── backtest/          # 回测引擎
├── strategies/        # 策略代码
├── fetch_data.py      # 数据脚本
└── 学习笔记/          # 知识积累

仓库里不能有的:
├── .env               # API key, 私钥
├── config.json        # 服务器 IP, 交易所配置
└── trade_logs/        # 实盘交易记录（可能暴露策略和仓位）
```

原则：**代码是武器，配置是弹药。武器可以分享，弹药必须私藏。**

### 5.2 .gitignore 不是可选的

我们现在的 `.gitignore`：
```
.env
```

但这还不够。将来实盘上线后需要加入：
```
logs/trades_*.csv       # 交易记录暴露仓位
config.prod.json        # 生产环境配置
*.key                   # 任何密钥文件
```
