# Day 1 学习笔记 — 数据管线

## 一、交易所 API 数据结构

### 1.1 OHLCV — 最基础也最容易被滥用

```
O = Open   (开盘价 — 这根 K 线开始时的第一笔成交价)
H = High   (最高价 — 这根 K 线期间的最高成交价)
L = Low    (最低价 — 这根 K 线期间的最低成交价)
C = Close  (收盘价 — 这根 K 线结束时的最后一笔成交价)
V = Volume (成交量 — 这根 K 线期间的总成交量，以 base currency 计)
```

**关键认知**：你在回测里用的 `close`，在实盘中只有这根 K 线走完的那一刻才存在。所以任何策略都不能在 K 线中间用收盘价做决策——这就是最常见的未来函数。

### 1.2 K 线周期对策略的深刻影响

| 周期 | 数据特点 | 策略含义 |
|------|---------|---------|
| 1m / 5m | 噪声主导，信号/噪声比极低 | 只有做市商和 HFT 有意义 |
| 15m / 1h | 有一些可交易的结构，但随机游走主导 | 趋势策略在这里大概率亏钱（我们的数据验证了） |
| 4h | 日内结构开始浮现 | 中频策略的甜点区 |
| **1d** | **趋势结构最清晰，噪声最低** | **我们的策略在这个周期有效** |
| 1w | 信号太少，一年才 52 根 | 只适合宏观判断 |

**核心洞察**：我们的 1h 回测全部灾难性亏损（夏普 -0.86 ~ -1.10），1d 回测全部正夏普（0.22 ~ 0.61）。这不是参数问题——是时间框架问题。**1h 级别的价格变动接近随机游走，技术指标在噪声中没有预测力。**

### 1.3 深度和成交数据的定位

- **深度（Order Book）**：看得见但摸不着——挂单可以随时撤
- **成交（Trades）**：真实发生了的交易——不会说谎

深度数据告诉我们「市场在演什么」，成交数据告诉我们「市场真正在做什么」。两者必须结合起来看。

## 二、数据清洗的六个环节

### 2.1 为什么清洗不是可选的

原始交易所数据有五种常见问题：

| 问题 | 怎么来的 | 不处理的后果 |
|------|---------|------------|
| 重复行 | 交易所 API bug / 拉取重试时重叠 | 同一根 K 线被回测算两次 |
| 时间戳不对齐 | 不同交易所的 K 线边界可能差几秒 | 交叉验证时匹配错位 |
| 缺失 K 线 | 交易所宕机、维护、API 限频 | 回测里的「连续」其实是假的 |
| 价格跳跃 | 插针、闪崩、交易所数据错误 | 策略把这当真信号 → 错误入场 |
| 成交量异常 | 洗盘、对倒、做市商刷量 | 量价分析被污染 |

### 2.2 五个步骤的深层逻辑

```
第1步 去重
  问题：币安 API 拉取时 batch 重叠 → 同一根 K 线出现两次
  后果：回测里策略看到两次同一根 K 线 → 可能重复入场
  做法：df[~df.index.duplicated(keep='first')]

第2步 时间戳对齐
  问题：币安的 12:00 K 线实际是 12:00:03，OKX 的是 12:00:00
  后果：两个数据源对比时匹配不上
  做法：统一 floor 到整点/整天（df.index.floor('h')）

第3步 缺失检测
  问题：预期 17544 根 K 线，实际只有 17520 根 → 少了 24 根
  根因：交易所宕机 / 维护 / 那段时间根本没有交易
  做法：生成完整时间序列，标记缺失项，缺失 > 5% 报警

第4步 异常值标记
  问题：某根 K 线涨跌幅 > 20%，或者成交量 > 均值 10 倍
  到底是真行情还是数据错误？
  做法：标记但不删除——留到策略层面判断。标记的 K 线默认不生成入场信号

第5步 交叉验证
  问题：币安 BTC 收盘价 65000，OKX 收盘价 65300 → 差 300U（0.46%）
  应该相信谁？
  做法：取均价作为「统一价格」，保留两个交易所的原始值。
        价差 > 0.5% 标记为数据分歧事件
```

### 2.3 为什么标记而不是删除

> 异常数据点不一定是错误数据。插针可能是真行情（有人被爆仓），放量可能是真突破。

删了 → 你这个时间点就白了一根 K 线 → 回测在这里「看不到」真实发生过的事。

标记 → 策略在做入场决策时**知道**这根 K 线不太对，选择跳过。但出场不受影响（不能因为数据异常就不止损）。

## 三、双源交叉验证：为什么至少两个数据源

### 3.1 单一数据源的盲区

只用币安的数据 → 你永远不知道：

- 币安的 K 线和其他交易所是否一致（交易所之间的价差）
- 币安的数据有没有被修正过（交易所偶尔会修正历史 K 线）
- 币安的成交量是不是洗出来的（小币种刷量严重）

### 3.2 交叉验证告诉你的事

```
价差 < 0.1% → 数据一致，可信
价差 0.1-0.5% → 正常偏差，取均价
价差 > 0.5% → 数据分歧，标记 + 报警
价差 > 2% → 可能一个交易所出问题了，不交易
```

我们的数据质量报告显示币安和 OKX 平均价差仅 0.03-0.05%，说明 Layer 1（BTC/ETH/SOL）的数据质量很高。如果将来做小币种，价差可能会大很多。

## 四、数据源分层策略

### 4.1 为什么分层

不是所有数据都值得实时拉取。分层 = 按重要性和成本分配资源。

```
Layer 1: 现货 K 线 + 深度（主信号）
  → 最高优先级，最高质量要求
  → 双源交叉验证，缺失 > 5% 报警
  → 已完成 ✅

Layer 2: 合约持仓量 OI + 资金费率（情绪）
  → 辅助信号，用来调节仓位
  → Day 11 接入
  → 频率：每小时更新

Layer 3: BTC.D + 恐惧贪婪指数（宏观）
  → 市场环境判断用
  → Day 11 接入
  → 频率：每天更新

Layer 4: 链上数据 + KOL 社交（归因）
  → 解释「为什么会涨跌」，不是预测「会不会涨跌」
  → Day 19-20 接入
  → 频率：链上每天，KOL 每小时聚合
```

### 4.2 分层的关键设计决策

**只有 Layer 1 的数据能触发交易信号。** Layer 2-4 的数据只做两件事：
- 调节仓位大小（市场环境不适宜 → 减仓）
- 解释复盘归因（这笔亏损是因为什么事件）

这个设计防止了一个常见错误：**用辅助数据替代主信号做决策**。KOL 喊单不是开仓理由，资金费率变化不是开仓理由——只有你的策略逻辑才是。

## 五、增量更新 vs 全量拉取

### 5.1 什么时候用哪个

```
增量更新 (update_data.py):
  场景：每日运行，保持数据最新
  逻辑：读已有数据的最新时间戳 → 只拉新的 → 拼接
  优势：快（几秒 vs 几分钟）
  限制：如果最新数据 > 7 天没更新，建议跑全量

全量拉取 (fetch_data.py):
  场景：首次使用 / 数据缺口太大 / 怀疑数据质量
  逻辑：拉取全部 2 年历史
  优势：完整，干净
  限制：慢，需要几分钟
```

### 5.2 数据连续性检查

增量更新最关键的一步：**新数据和旧数据之间有没有缺口？**

```python
gap = new_earliest - old_latest
if gap > expected_gap * 3:  # 超过 3 个周期的缺口
    warning("数据缺口！")
```

缺口可能意味着：交易所宕机、API 限频导致漏拉、程序异常中断。缺口小时可以忍，缺口大了必须跑全量。

## 六、Parquet vs CSV

### 6.1 我们的存储选型

```
raw/    → CSV     (人类可读，方便 debug，可以 Excel 打开)
clean/  → Parquet (压缩率高 10x，读写快 10x，列式存储)
```

CSV 存原始数据的原因：拉下来什么样就什么样，不改。出了问题可以追溯到原始数据。

Parquet 存清洗后数据的原因：回测需要快速读入 17520 行 × 多列，Parquet 列式存储比 CSV 快得多。

### 6.2 不要用 SQLite 存大量时间序列

SQLite 适合存元数据（data_catalog.json / fetch_log.json），不适合存 K 线。原因：
- 时间序列查询效率不如 Parquet（列式存储 vs 行式存储）
- 单文件大数据库容易损坏
- 不如 Parquet 方便做 pandas 操作
