# Day 5 学习笔记 — 模拟盘验证

## 一、模拟盘存在的意义

### 1.1 回测 ≠ 实盘

回测和实盘之间的差距来自几个硬性原因：

| 差距来源 | 回测中的表现 | 实盘中的现实 |
|---------|------------|-----------|
| 成交价格 | 信号价 ± 固定滑点 | 真正的 order book 深度决定成交价 |
| 信号时间 | K 线收盘后立即执行 | 有网络延迟、API 延迟 |
| 流动性 | 无限流动性假设 | 你的单子本身会推动价格（大单） |
| 市场冲击 | 忽略 | 大单对订单簿的冲击 |
| 限价单成交 | 假设价格到就成交 | 排队、部分成交、滑点 |

模拟盘用模拟的真实条件填补这些差距——在不动真实资金的情况下暴露问题。

### 1.2 模拟盘的三个核心问题

1. **信号是否在正确的时机生成？** — 没有未来数据泄漏
2. **实际成交价和信号价偏差多少？** — 滑点模型是否准确
3. **极端行情时信号和行为是否正常？** — 闪崩/暴涨时不崩

## 二、设计思路：抽象接口 + 可插拔实现

### 2.1 为什么不直接接币安 testnet

- 币安 testnet 需要 API key + 网络配置，在国内不稳定
- 本地历史数据回放更可控——可以先验证策略逻辑，再验证执行
- 抽象接口设计可以在两种模式间无缝切换

### 2.2 两个抽象层

```
MarketDataFeed（行情数据源）
├── HistoricalDataFeed   → 本地 parquet 逐行 yield（当前）
└── BinanceTestnetFeed   → 币安 testnet WebSocket（将来）

ExecutionSim（成交模拟）
├── LocalExecutionSim    → CostModel 驱动模拟成交（当前）
└── BinanceTestnetExecution → 币安 testnet 订单撮合（将来）
```

切换数据源和成交模拟，PaperTrader 核心调度逻辑不动。

### 2.3 PaperTrader 调度流程

```
1. 从 MarketDataFeed 取下一根 K 线
2. 调用策略 on_bar() 生成信号（和回测完全一样的逻辑）
3. 信号传入 ExecutionSim 模拟成交
4. 记录成交价/成本/滑点
5. 更新净值
6. 循环 → 结束后输出偏差分析
```

## 三、模拟盘 vs 回测：BTC 1h 实际对比结果

用同样的 EMA 20/50 趋势策略，跑 17,520 根 K 线（2024.07-2026.07）：

| 指标 | 回测 | 模拟盘(市价单) | 差异 |
|------|:---:|:---:|:---:|
| 总收益率 | -76.52% | -80.56% | **-4.04%** |
| 交易笔数 | 268 | 268 | 0 |
| 入场平均滑点 | 5.0 bps（假设） | 13.5 bps（实际） | +8.5 bps |
| 错失信号 | 0 | 0 | — |

**关键发现**：
- 净值偏差 -4.04% 来自成本模型的差异——回测用的 0.05% 固定滑点太乐观了
- 模拟盘的真实滑点约 13.5 bps（双向 27 bps），是回测假设的 2.7 倍
- 信号一致性 100% — 模拟盘产生的信号和回测完全相同（都用同一策略逻辑）
- 判定：🔶 中度偏差 — 成本模型需要校准

### 3.1 限价单 vs 市价单

| 指标 | 市价单 (Taker) | 限价单 (Maker) |
|------|:---:|:---:|
| 交易笔数 | 268 | 259 |
| 成交率 | 100% | 98.3% |
| 总收益率 | -80.56% | **-79.51%** |
| 平均滑点 | 13.5 bps | 12.5 bps |
| 错失信号 | 0 | 9 |

限价单收益率高出 +1.04%，代价是错失 9 个信号（3.4%）。对于 BTC 1h 策略，**限价单更优**。

## 四、偏差分析框架

### 4.1 四个对比维度

1. **信号一致性** — 同一 bar 是否产生相同的 entry/exit 信号？不一致说明回测逻辑和模拟执行有差异
2. **成交价偏差** — 模拟成交价和回测理想价的差距。差距大说明滑点模型不准
3. **滑点分布** — 模拟盘的实际滑点范围和分布。跟回测假设（固定值）对比
4. **净值偏差** — 最终收益差异。这是所有偏差的汇总

### 4.2 偏差可接受范围

| 净值偏差 | 判定 |
|---------|------|
| <2% | ✅ 回测可信 |
| 2-5% | ⚠️ 轻微偏差，关注滑点模型 |
| 5-10% | 🔶 中度偏差，需调成本参数 |
| >10% | 🔴 严重偏差，检查未来函数或成本模型 |

## 五、核心教训

### 5.1 回测的滑点假设太乐观是最大的坑

回测默认 0.05% 固定滑点，但真实世界（即使是 BTC）的平均滑点约 0.135%。这个 2.7x 的差距在 268 笔交易叠加后就是 4% 的净值差异。

### 5.2 模拟盘不是"可选的验证步骤"

它是在不动真金白银的情况下，暴露你回测假设和真实世界差距的唯一方式。如果跳过模拟盘直接上实盘，你的第一个发现（"怎么亏得比回测多"）已经花了真钱。

### 5.3 抽象接口的价值不在当下，在扩展性

现在只用本地历史数据，但接口设计保证了将来只需要写一个新的 `BinanceTestnetFeed` 实现，就能切换到实时 testnet 数据，不会动核心逻辑。
